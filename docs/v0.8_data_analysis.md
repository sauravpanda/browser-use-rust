## The three real gaps to upstream

### 1. Calibration (10pp opportunity, biggest single lever)
- Upstream: 77.0% done-correctness (when it claims success, it's right 77% of the time)
- Best of ours (v0.8.2): 67.2%
- **Math: if we matched upstream's calibration with our current self-report rate, we'd land at 194 × 77% = 149 / 198 = 75.3% judge.** That's above upstream's 72.7%.
- This is the *single highest-value lever* and explains the ENTIRE judge gap on its own.

The existing 2-pass validation prompt (`_VALIDATION_CHECKLIST` in agent/__init__.py:178) has the right shape but isn't strong enough — the LLM says "yes I'm sure" then commits the same wrong answer.

### 2. Step count (2.2× more steps than upstream)
- Upstream: 16.6 mean steps/task (median much lower since some tasks need 40+)
- Ours: 26-39 steps/task
- We're churning 2× the work to get *worse* answers
- Drives both cost (2.5× higher) and likely calibration (more turns = more chances to drift from the right page/answer)

### 3. Action errors (9× upstream)
- Upstream: 22 total action errors across 198 tasks (~0.11/task)
- Ours: 115-205 (~0.6-1.0/task)
- We have 5-9× more execution failures during navigation/clicks
- Sources possibly: stale-index errors, click-on-wrong-element, dropdown handling, captcha re-tries

## Why prior cost-optimization phase failed

Looking at v0.8.1 → v0.8.7, every "improvement" cost judge points:
- v0.8.1 (anchor): 67.1%
- v0.8.2 (history window 6→3): 66.2% (-0.9pp)
- v0.8.4 (crash recovery + LLM timeout): 64.6% (-1.6pp)
- v0.8.5 (loop sig norm): 61.6% (-3pp, reverted)
- v0.8.7 (LLM retry/timeout parity): 65.2% (-1.9pp from anchor)

We were optimizing the wrong axis. Each defense layer caught real failures but the recovered tasks finished in degraded states the judge rated *worse* than the natural failures. We weren't moving the dominant lever (calibration).

## Plan: three-phase, data-driven

### Phase A — Calibration (highest EV, target: +5-10pp judge)

**v0.8.8 — Mandatory evidence quoting in validation prompt.** Modify `_VALIDATION_CHECKLIST` to require: "For EACH claim in your answer, quote the literal page text supporting it. If you cannot find a quote, the claim is unverified — fix the answer or do another extract." Strong specific signal beats "are you sure?".

**v0.8.9 — Force fresh extract before done.** If the agent hasn't called `extract_structured_data`, `page_text`, or `get_text` in the LAST 2 TURNS before its done call, mechanically inject one before the validation prompt fires. Closes the "agent answers from memory of an old page" failure mode. (Detection in agent/__init__.py: peek at `self._recent_tool_names` window before triggering validation.)

**v0.8.10 — Doubt-listing variant.** Replace the "are you sure?" check with "list 3 things you're uncertain about in this answer; for each, decide whether to verify or accept the uncertainty." Forces the LLM to surface its own doubts instead of asserting confidence.

Test each in isolation. Pick the winner; combine the top 2 for v0.8.11.

### Phase B — Step efficiency (target: -30% step count, modest judge gain)

**v0.8.12 — Force extract within N steps of arriving on a page.** Track per-tab/URL "steps since landing"; if it exceeds 4 with no read-tool call, inject "you've explored this page for 4 turns without reading content — extract or move on."

**v0.8.13 — Prompt restructure to discourage scroll-then-scroll.** Right now the LLM scrolls a lot looking for things. Add explicit instruction: "Do not scroll more than once before extracting — if scroll-1 didn't reveal what you need, the content is probably not on this page or needs a different approach."

### Phase C — Action error reduction (target: -50% errors, secondary judge gain)

**v0.8.14 — Categorize and target.** First pull error breakdown from the trace data (`error_log` field is in completeHistory, which I left out of cache — need to refetch with that). Identify top-3 error types. Likely candidates:
- Stale-index (already heavily defended; may still be top)
- Click-on-non-clickable (we emit static text now via index=0, but maybe LLM still tries to click them)
- Captcha encounters (we have a 5s wait, but many fail anyway)
- Dropdown/select errors

### Phase D — Validate composability

**v0.8.15+ — Combine winners from A+B+C.** Verify gains don't cancel.

## Notes on what NOT to do

- Don't keep adding defense layers (v0.8.3/v0.8.4/v0.8.7). They demonstrably cost judge.
- Don't bundle multiple changes in one version. Phase 1's bundling cost us interpretability.
- Don't optimize cost first — calibration drives more value than the cheapest version we could ship.

## Open questions worth pulling more data for

1. **Which 65 tasks does the judge reject?** Need per-task judge verdicts (separate endpoint or DB query). Without this, we're speculating about failure modes.
2. **What error types account for our 205 action errors?** Need to refetch with `completeHistory` to enumerate.
3. **Does the per-task step distribution have a fat tail?** If 80% of tasks finish in 15 steps and 20% in 80+, that 20% is the cost driver.
4. **Are upstream's 11 self-reported failures legitimate honesty (it knew it failed) or different mechanic (it crashed before calling done)?** Compare the histories.
