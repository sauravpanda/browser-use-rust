# Experiments to re-try & open issues — post-v0.8.22

Now that the foundational issues are addressed (thinking_level honored,
DOM walker not over-bloated, context bloat capped, persistent state
contract enforced, real CDP send_keys, Laminar tracing, accurate
token/cost accounting), several previously-discarded experiments
deserve a second look. Their failures may have been masked by — or
caused by — the broken substrate.

This file also tracks **open infrastructure issues** that aren't
single experiments but block real progress (Rust-side crashes being
the biggest one).

## Critical open infrastructure issue

### Rust-side crashes ("Stage errors: run_agent:")

**Status**: still happening intermittently as of v0.8.22 despite three
defense layers (v0.8.3 top-level wrapper, v0.8.4 per-step LLM/tool
try/except, v0.8.7 retry/timeout parity).

**Symptom in eval logs**: `Completed stages: setup_browser | Stage errors: run_agent: <Exception>`.
The eval framework records the task with no answer → judge counts as
0% → asymmetric pull on judge_pass_rate.

**Hypothesis (unverified)**: the residual cases are likely Rust-side
crashes (panics, CDP socket death producing typed errors we don't
guard, or process-level kills like SIGSEGV/OOM that no Python
try/except can catch). Crates with `unwrap()` / `expect()`:
- `crates/bu-browser/src/lib.rs` — many CDP response unwraps
- `crates/bu-cdp/src/*` — socket layer

**To diagnose**: capture the FULL exception text from the eval log
(class + first 200 chars of message). The class tells us whether it's:
- `TimeoutError` → step itself hangs, need per-step timeout
- `ConnectionError` / `BrokenPipeError` → CDP socket died
- `BrowserError::*` → Rust raised typed error, specific path to guard
- `RuntimeError` / `AttributeError` → Python bug in our code
- No traceback / process exit → Rust panic or OS kill

**Fix surface**:
- Subscribe to CDP `Target.detachedFromTarget` (existing task #115)
- Convert Rust `unwrap`/`expect` to typed `BrowserError` returns
- Surface socket failures as recoverable errors rather than letting
  downstream calls panic
- Possibly add per-step timeout (currently only per-LLM-call and
  per-tool-call have timeouts)

This is foundational work, not a behavior change. Belongs in v0.9.0.

## Experiments to re-try

### 1. Loop signature normalization (was v0.8.5 — reverted)

**Original**: dropped `index` and `selector` from tool args before
hashing the loop-detection signature. So `click(idx=5) → scroll →
click(idx=12)` would collapse to one signature and trigger
`LOOP_DETECTED` after 3 occurrences.

**Why discarded**: -3pp judge / +20% cost vs v0.8.4. False-positive
nudges fired on legit list-iteration patterns (clicking through search
results), bloating context with `[LOOP_DETECTED]` UserMessages and
confusing the LLM.

**Why retry**: with v0.8.22's medium thinking actually engaged + v0.8.20's
context bloat fixes, the agent should:
- Make better decisions per-step (less likely to actually loop)
- Be less confused by spurious nudges
- Not pay the same context-bloat cost (we cap collapsed history now)

**How to retry**: ship as a v0.8.x with REPEAT_THRESHOLD raised from 3 → 5,
and COOLDOWN_STEPS from 3 → 6 (codex's earlier suggestion). Less
aggressive trigger, gives the agent room to do legit batching.

**Risk**: medium. If false-positives still bite, revert.

---

### 2. Dead-index nudge prompt clause (part of v0.8.10 — reverted in v0.8.13)

**Original**: prompt addition "After 2 retries on the same failed index,
switch tools or navigate elsewhere — do not loop on a dead index."

**Why discarded**: caused over-rotation. Agent abandoned working flows
on every minor stale-index event. -2.88pp judge.

**Why retry**: now that v0.8.22 has medium thinking + v0.8.21 has cleaner
DOM (no KEEP_ATTRS bloat), the agent might handle the nudge with more
nuance instead of treating it as a hard "give up" signal.

**How to retry**: re-add the clause but soften — change "switch tools or
navigate elsewhere" to "consider whether you've already tried this
approach; if you haven't read content recently, extract or pick a
different element from the fresh snapshot."

**Risk**: medium-high. Was the worst single regression in the v0.8.x
line. Only retry after v0.8.22's improvements are confirmed solid.

---

### 3. LIVE / TIME-SENSITIVE CONTENT prompt (part of v0.8.11 — reverted in v0.8.14)

**Original**: "if your first navigation choice is a direct URL guessed
from training memory, prefer clicking through the site's own UI."

**Why discarded**: codex hypothesis was that this caused wandering.
v0.8.11 dropped 14pp from baseline.

**Surprise from data**: v0.8.14 (which removed this) was -4pp WORSE
than v0.8.13 (which kept it). So removing this clause HURT, not helped.

**Why retry**: the clause was probably right; the v0.8.11 regression was
from something else (mechanical downgrade interaction or LOCATE clause).
Add it back ALONE on a clean baseline to isolate.

**Risk**: low-medium. Worth a single-change re-test.

---

### 4. LOCATE-THEN-EXTRACT prompt (part of v0.8.11 — reverted in v0.8.14)

**Original**: "when the task names a specific section AND the page has
multiple distinct sections, use search_page(pattern) to find the right
section first, THEN extract."

**Why discarded**: bundled with #3 in the v0.8.11 regression. Codex
flagged it as the more likely culprit (extra search_page calls + scroll).

**Why retry**: similar reasoning to #3 — re-test alone, on the better
substrate.

**Risk**: medium. Could add steps. But the underlying problem (NASA
"U.S. commercial companies" generic vs upstream's "CSA, SpaceX")
remains real.

---

### 5. Higher REPEAT_THRESHOLD on existing loop detector

**Not previously tried**. The current strict-args loop nudge fires at
threshold 3 in a 6-turn window. With v0.8.22's better thinking, the
agent should less commonly hit even strict loops; raising to 4 or 5
might reduce false-fires without missing real loops.

**Risk**: low. Pure tuning.

---

### 6. KEEP_ATTRS expansion — narrower retry

**Original (v0.8.17)**: added 22 new attrs. Caused -3pp regression
because it bloated dense-page DOM.

**Why retry**: filter UIs DO need the constraint info. Try a much
narrower list — just `min`, `max`, `step` for range/date inputs (the
specific case where we lose information). Skip the aria-* and
multiple/data-value/etc. that contributed most of the noise.

**How to retry**: 3 attrs not 22. Test in isolation.

**Risk**: low if kept narrow. Was the cause of v0.8.21's recovery.
