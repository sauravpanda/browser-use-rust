# v0.12.0 Plan — Structural Rewrite for Cost Reduction

**Status:** v0.12.x patch series in progress; current focus is cost without regressing blocked-site recovery.
**Last updated:** 2026-05-23

## Current 0.12.x Patch Plan

The 0.12.8-0.12.10 work lowered cost incrementally, but the main remaining
gap versus Python is still per-step prompt size: Rust is around 12.4k prompt
tokens/step while Python is around 5.6k prompt tokens/step.

### v0.12.11 result

Run `kh7eqafkbmkmh55nj3hmps86t5874gpy` made the agent cheaper by ending
blocked/search-loop tasks earlier, but the success tradeoff was too large:

- Success fell from 149/198 in 0.12.10 to 133/198 in 0.12.11 (-8.08pp).
- Avg cost fell from `$0.054125` to `$0.047429` (-12.37%).
- P90 steps fell from 34 to 26.
- Give Up failures rose from 35 to 51.
- Prompt tokens per step stayed flat at about 12.4k, so the cost win came from
  shorter runs, not from cheaper turns.

### v0.12.12 adjustment

Ship a narrow correction before continuing the deeper prompt cleanup:

- Keep the shared blocked-site policy and anti-loop nudges from 0.12.11.
- Restore the softer 0.12.10 blocked/search force-final thresholds:
  blocked window 8, force at 5 blocked states after step 15; search fallback
  window 8, force at 6 search states after step 20.
- Allow bounded external-result completion when all target-site variants are
  blocked and a visible result title/snippet/URL directly answers a static
  public fact.
- Continue rejecting snippet-only completion for live/current data, prices,
  availability, bookings, account-gated pages, locators, or actions that must
  be performed on the site.

Expected effect: recover a meaningful share of the 27 pass-to-fail regressions
from 0.12.11 while preserving some of the step/cost tail reduction.

### v0.12.12 result

Run `kh7fwtazjq9bc9d2s40wtbgqsx8740cw` recovered part of the 0.12.11
success regression, but gave back the cost win:

- Success improved from 133/198 in 0.12.11 to 140/198 (+3.54pp), slightly
  above the Python baseline of 138/198.
- Avg cost rose from `$0.047429` to `$0.055607`, slightly worse than 0.12.10
  (`$0.054125`).
- Avg steps rose from 14.40 to 15.38, still lower than 0.12.10's 16.49.
- P90 steps stayed improved at 28 versus 34 in 0.12.10.
- Prompt tokens per step worsened to about 13.1k, so the next slice should
  remove redundant finalization turns rather than add more prompt policy.

Trace detail showed 100/198 tasks called `done` twice and 102/198 had a
provisional finalization turn. Those provisional-final tasks averaged about
`$0.0614` versus `$0.0487` for the rest.

### v0.12.13 adjustment

Disable global self-validation by default while keeping the targeted
final-answer recovery nudges and list-count guard:

- Default `Agent(..., self_validate=False)`.
- Preserve explicit `self_validate=True` for callers that want the old
  confirmation-turn behavior.
- Keep task-specific recovery nudges before finalization.
- Keep `_done_count_check_message` so underspecified top-N/list answers still
  get one correction turn.

Expected effect: remove one LLM round trip from many successful tasks, reducing
steps and completion/prompt cost without touching browsing behavior.

### v0.12.13 result

Run `kh70ap03gg75bhzqvb2zek6ss1876dc8` proved the finalization cost lever,
but the accuracy loss was too large:

- Success fell from 140/198 in 0.12.12 to 132/198 (-4.04pp).
- Avg cost fell from `$0.055607` to `$0.046901` (-15.66%).
- Avg steps fell from 15.38 to 13.63.
- P90 steps fell from 28 to 26.
- Incorrect Result failures rose from 16 to 24.

The change worked mechanically: validation prompt turns went to zero, and
double-`done` tasks fell from 100/198 to 1/198. The removed validation was
catching real bad finals, so fully disabling it is not a good baseline.

### v0.12.14 adjustment

Use selective lightweight validation instead of global-on or global-off:

- Default `Agent(..., self_validate=True, self_validate_risk_only=True)`.
- Validate high-risk finals only: search/filter/sort/locator tasks, counted
  top/first/list tasks, recency/current/latest tasks, price/date/availability
  tasks, and blocked/external-evidence finals.
- Skip validation for low-risk static summaries.
- Shorten the validation prompt to a compact three-point check.
- Preserve `self_validate_risk_only=False` for callers that want the old
  validate-every-final behavior.

Dry-run over the 0.12.13 traces estimates validation would apply to about
73 successful step>=3 tasks, rather than the 102 provisional finalization
turns observed in 0.12.12.

### v0.12.14 result

Run `kh7e5mp7140yy41e4kzb5801sh8798am` recovered much of the 0.12.13
accuracy loss while preserving most of the cost win versus 0.12.12:

- Success improved from 132/198 in 0.12.13 to 137/198 (+2.53pp).
- Avg cost rose from `$0.046901` to `$0.050439` (+7.54%).
- Avg steps rose from 13.63 to 14.69.
- P90 steps stayed flat at 26.
- Failure categories improved from Give Up 42 / Incorrect Result 24 to
  Give Up 39 / Incorrect Result 21 / Step Limit 1.

The change worked mechanically: double-`done` tasks landed between the two
prior extremes:

- 0.12.12: 100 tasks had 2+ `done` calls.
- 0.12.13: 1 task had 2+ `done` calls.
- 0.12.14: 60 tasks had 2+ `done` calls.

Token analysis against the Python baseline showed the next bottleneck is the
static Gemini prefix. Rust's cache hit rate is high, but it is high because
the prompt carries a large repeated system/tool prefix:

- Python baseline: avg input 67.9k, cache read 11.2k, fresh input 56.7k.
- Rust 0.12.14: avg input 187.1k, cache read 118.5k, fresh input 68.5k.
- Rust's cached-token share was 63.4%, but cached Gemini tokens are still
  billed at the discounted cache rate.
- Per-step Rust metadata showed `prompt_tools_bytes=33,114` and
  `prompt_system_bytes=5,606`; alias tool declarations alone were about
  10.6KB of duplicated schema every LLM call.

### v0.12.15 adjustment

Trim the default advertised tool schema without removing runtime
compatibility:

- Keep upstream compatibility aliases in `tools_by_name` so old/custom alias
  calls still dispatch.
- Hide alias declarations from the default LLM tool schema.
- Add `Agent(..., expose_tool_aliases=True)` to restore the old full schema.
- Update prompts to prefer canonical names such as `send_keys` rather than
  alias names such as `press_keys`.

Measured locally, default LLM tool declarations drop from 62 tools /
33,114 bytes to 44 tools / 22,456 bytes, a 32% schema-byte reduction. This
mostly attacks cached-prefix cost and first-call fresh prompt size; it is
expected to be lower-risk than removing canonical tools.

- v0.12.16: zero-call canonical tool trim or per-task tool profile.
- v0.12.17: memory/read-state prompt consistency.
- v0.12.18: task-specific guidance dedupe.
- v0.12.19: DOM/result-card policy refinement.

## Option 1 — DOM relevance ranking + budget cap

### Hypothesis
Per-step DOM snapshot is 50-65 KB (vs upstream ~30-35 KB). Median step has ~13-16k tokens of uncached input from the snapshot alone. Cutting per-snapshot bytes by ~30% via prioritization + budget cap should drop median cost ~6-10% with no accuracy loss, IF prioritization preserves the elements the agent actually picks.

### Spec

**1. Element scoring in `crates/bu-dom/src/script.js`** — emit a per-element relevance score during the walk. Inputs:
- **In-viewport bonus** (already implicitly preferred via early DOM walk order, but make explicit): +5
- **Region** (closest ancestor `<header>/<nav>/<main>/<aside>/<footer>` or `[role=...]`): main +3, header +2, nav +2, footer −2, aside −1
- **Element type strength**: `<a href>` +3, `<button>` +3, `[role=button]` +2, `<input>` +3, `[onclick]` +1, `[cursor:pointer]` +0
- **Has accessible name**: text/aria-label/title +2, has only icon/svg child (no name) +0
- **Position-in-flow**: top of document +1, deeply nested +0
- **Static text**: leave as-is, do not score (already culled to narrow tag set)

**2. Top-N budget cap in `to_llm_string()`** — emit interactive elements in `score DESC, tree-order ASC` order, cut at element count = 80 (or byte cap = 35,000, whichever fires first). Static text emitted normally as currently. Below-cut elements available via scroll → re-snapshot.

**3. Prompt addendum** — single paragraph noting the snapshot is "ranked by relevance, top-N visible" so the agent doesn't assume it's seeing everything. Include scroll-to-find guidance.

**4. v0.10.4 instrumentation** — add per-call `elements_emitted` count + `elements_culled` count to detect over-aggressive culling.

### Risk profile
- **HIGH on accuracy**: scoring is heuristic; wrong score → wrong elements → wrong actions. v0.11.0 (40k DOM cap), v0.11.6 (text trim), v0.11.8 (paint-order culling on text) all failed via similar mechanisms.
- **Mitigation 1**: ship behind env-var flag (default OFF); enable only via eval config until validated.
- **Mitigation 2**: cap at 80 elements only — most pages have ≤80 visible interactive elements anyway, so cap rarely fires on simple pages.
- **Mitigation 3**: log culled elements for first 10 eval tasks; manually inspect to confirm cut content is genuinely low-relevance.

### Estimated impact (per cross-experiment subagent analysis)
- Cost: −5-10% per task (median attack)
- Accuracy: tail risk — could lose 3-5pp if scoring is bad
- Tokens: −15-25% per snapshot on dense pages

### Effort
3-5 days:
- Day 1: scoring algorithm in script.js + tests on synthetic DOMs
- Day 2: budget cap + tree-order tiebreak + emit
- Day 3: prompt update + env flag + instrumentation
- Day 4-5: eval-validate, tune scoring weights

## Option 2 — HistoryItem + LLM summarization (USER-PREFERRED)

### Hypothesis
Conversation history grows linearly with steps. Even cached, every step pays `cache_read` on the entire prior history. A 20-step task accumulates ~50-100k tokens of message history. Replacing accumulated raw `(AssistantMessage, ToolResultMessage*)` pairs with a compacted LLM-summarized "memory" string would:
- Cut steady-state cached prefix bytes ~50-70%
- Cut cache_creation cost on the step where summarization fires
- Preserve agent context via structured memory + recent-step retention

This is the architecture upstream Python `browser_use` uses (see `browser_use/agent/message_manager/service.py` and `views.py::HistoryItem`).

### Spec

**1. New `HistoryItem` struct** in `python/browser_use_rs/agent/__init__.py` (or new `_history.py`):
```python
@dataclass
class HistoryItem:
    step: int
    evaluation: str            # from <evaluation_previous_goal>
    memory: str                # from <memory>
    next_goal: str             # from <next_goal>
    actions: list[ActionRecord] # tool calls + their long_term_memory
    error: str | None          # if step had errors
```

Each step produces ONE HistoryItem (not the current 1 AssistantMessage + N ToolResultMessage pairs in self._messages).

**2. ActionRecord** — replaces full ToolResultMessage content with:
```python
@dataclass
class ActionRecord:
    tool_name: str
    args_summary: str       # short repr of args (e.g., "click([5])")
    long_term_memory: str   # 1-line summary suitable for history context
    extracted_content: str | None  # full content, only kept if include_in_memory=True
```

**3. Message-list construction at LLM call time** — replace current accumulating `self._messages` with:
- System prompt (cached)
- Initial task UserMessage (cached)
- `<agent_history>` block in the per-step state UserMessage, built from compacted HistoryItems
- Recent N items kept verbatim (last 3-5 steps)
- Older items: LLM-summarized into a single `<compacted_memory>` string

**4. Compaction trigger** — fire LLM summarization when:
- History char count > 40,000 (~10k tokens)
- AND last summarization was ≥6 steps ago (avoid thrash)

**5. Compaction LLM call** — separate cheap LLM call (gemini-3-flash-minimal) given:
- Current `<compacted_memory>` string
- Items 0..N-keep_last from history
Return: new `<compacted_memory>` string (≤6000 chars).

**6. Tool result lifecycle change** — large extract / page_text results no longer go directly into ToolResultMessage. Instead:
- `extracted_content` stored on the HistoryItem with `include_in_memory=False`
- The CURRENT-step state UserMessage gets a `<read_state>` block with the just-completed extracts
- Next step: `<read_state>` cleared
- This is the upstream `include_extracted_content_only_once` pattern done correctly (the v0.11.2 attempt failed because it was bolted on without the HistoryItem foundation)

### Risk profile
- **MEDIUM on accuracy**: structured memory + LLM summary should preserve agent state better than the current native message list does for long tasks
- **HIGH on implementation complexity**: this is a real architectural rewrite
- **Cache-shape risk**: any change to message structure shifts the cached prefix → one-time cache rebuild per session amortized; but per the v0.11.20 lesson, the LLM may notice the structural change and react
- **Compaction LLM cost**: each summarization call is ~$0.001-0.005. Fired at most every 6 steps means ~3-4 calls per 20-step task = ~$0.003-0.020/task overhead. Net savings only if accumulated history savings > overhead.
- **Mitigation 1**: instrumentation. Log per-call: chars saved by compaction, summarization LLM cost, accuracy on tasks that crossed the compaction threshold vs those that didn't.
- **Mitigation 2**: feature flag (env var `BU_RS_HISTORY_COMPACTION=1`). Default OFF. A/B in eval.

### Estimated impact
- Cost: **−15-30% per task** on long tasks (≥15 steps) — the median task IS long enough to benefit
- Tokens: **−40-60% on cached prefix** at steady state
- Accuracy: should be neutral or slight positive (better-structured memory than raw conversation)

This is the **biggest cost lever remaining** if implemented correctly. Per user's intuition: "Option 2 might be biggest cost saver tbh."

### Effort
5-10 days:
- Day 1-2: HistoryItem + ActionRecord dataclasses, refactor `_run_tool` to populate them
- Day 3: replace `_messages` builder to construct from HistoryItems
- Day 4: implement compaction trigger + summarization LLM call
- Day 5: ToolResult lifecycle (move extracted_content to HistoryItem)
- Day 6-8: eval-validate, tune compaction thresholds
- Day 9-10: bug-fixes, edge cases (resume from injected state, error propagation)

## v0.12.0 success criteria (variance-aware)

Per the v0.11.x learnings: ±3pp accuracy / ±10% cost are noise floor on single-run evals. Real signals require ≥5pp accuracy or ≥15% cost moves.

### α / β gates (refactor ships — should be no-ops)
- ✅ Accuracy within ±3pp of v0.11.24's 73%
- ✅ Cost/task within ±10% of v0.11.24's $0.0689
- ✅ cost/M tokens within ±5% of v0.11.24's $0.289
- ✅ No crashes, no system_prompt_hash drift
- If a refactor ship moves the eval beyond noise, the bug is structural — fix BEFORE proceeding. Do not bundle into the next slice.

### v0.12.0 final gates (compaction live)
- ✅ Cost/task ≤$0.060 (−13% vs v0.11.24's $0.0689) on full WebBench_READ_v5
- ✅ Accuracy ≥70% (within ±5pp of v0.11.24's 73%)
- ✅ cost/M tokens ≤$0.270 (vs v0.11.24's $0.289 — confirms cache hit-rate improvement)
- ✅ Time/task ≤80s (vs v0.11.24's 69.6s — accept some latency for compaction calls)
- ✅ No crashes, no major action error spike
- ✅ v0.10.4 instrumentation shows: compaction fires N times per task with ≤1k token summary, no system_prompt_hash drift

If all gates pass, ship v0.12.0 as new canonical baseline. If 1-2 gates fail but trend is right, tune. If 3+ gates fail, the architecture port doesn't work on this substrate — accept v0.11.24 as the locked product baseline and pivot.

## Variance-aware bisect rules (locked workflow)

From the v0.11.x cycle, applies to all v0.12.x experiments:
1. ONE isolated change per ship. Bundles produce false attribution.
2. **cost/M tokens** is the only clean signal across runs. Track it primarily.
3. Treat <10% cost/task and <5pp accuracy moves as noise.
4. Look at MEDIAN, not just mean — tail tasks distort means.
5. After 3 consecutive ships in noise, the lever is exhausted; pivot.
6. Comment out reverted changes with clear `vX.Y.Z reverted because…` blocks.
7. Hash drift detection (`_check_cache_stability`) and per-call metrics from `_compute_call_metrics` are mandatory; don't disable them.
